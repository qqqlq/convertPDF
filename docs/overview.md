# SlidePDF 概要・開発経緯

## このツールは何をするもの？

SharePoint などで配信されている授業動画を見ながら、**スライドや板書を自動でキャプチャして PDF にまとめる** Chrome 拡張機能。

手動でスクリーンショットを撮る必要がなく、録画開始〜停止の間に画面に表示された内容を自動で収集し、重複を削除した上でPDFとして書き出す。

---

## 全体の流れ

```
動画を再生しながら録画開始
    ↓
スライドが切り替わる・板書が進む → 自動でキャプチャ
    ↓
録画停止 → 「PDFを生成」ボタン
    ↓
重複しているフレームを自動で削除
    ↓
slides.pdf としてダウンロード
```

---

## キャプチャのしくみ

スライドの変化を見逃さないため、3つの方法を組み合わせている。

### ① 差分検知（メイン）

250ミリ秒ごとに画面を確認して「前回と変わった？」を判定する。

```
前のフレーム        今のフレーム
┌──────────┐      ┌──────────┐
│ スライドA │  →  │ スライドB │  変化大 → キャプチャ！
└──────────┘      └──────────┘

┌──────────┐      ┌──────────┐
│ スライドA │  →  │ スライドA │  変化なし → スルー
└──────────┘      └──────────┘
```

「変化した」の判定：全ピクセルのうち 1.5% 以上の色が変わったら発火。

### ② 定期キャプチャ（保険）

板書のように**じわじわ変化する**コンテンツは差分検知だけでは撮り漏らすことがある。
そのため 10 秒ごとに必ずキャプチャする（5〜120 秒で変更可能）。

### ③ 手動キャプチャ

popup の「今すぐキャプチャ」ボタンで任意のタイミングにキャプチャできる。

## 重複削除のしくみ

キャプチャをそのまま全部 PDF にすると、ほぼ同じ画像が大量に並んでしまう。
そこで PDF 生成時に自動で重複を削除している。

### ステップ1: dHash（見た目の近さを数値化）

画像を 9×8 の極小サイズに縮小して「隣のピクセルと比べて明るい/暗い」を 64 個の 0/1 で表現する。
2枚の画像の差が 8 以下 → ほぼ同じ画像 → 1枚に統合。

### ステップ2: スーパーセット判定（追記の検出）

板書などでは、**同じ黒板に少しずつ書き足していく**ため、見た目は変わっていても
「別のスライド」ではなく「同じスライドの続き」のことが多い。

```
フレームA（古い）   フレームB（新しい）
┌──────────┐       ┌──────────┐
│ 数式①    │       │ 数式①    │  ← 保持されてる
│          │  →    │ 数式②    │  ← 追記された
│          │       │ 数式③    │  ← 追記された
└──────────┘       └──────────┘

「古いフレームのコンテンツが新しいフレームでも残っている」
→ 追記されただけ = 同じスライド → フレームBだけ残す
```

これを判定するために「コンテンツ保持率」を計算している。

### コンテンツ保持率の計算方法

ポイントは**「背景色を除いてコンテンツだけを比べる」**こと。

```
1. 古いフレームの全ピクセルの輝度（明るさ）を調べる
2. 輝度の中央値 = 「最も多い色」= 背景色として推定
   （緑黒板なら中央値 ≈ 88、白スライドなら中央値 ≈ 240）
3. 背景輝度から 40 以上外れたピクセル = コンテンツ
   （白チョーク: 輝度200 → |200-88|=112 → コンテンツ）
   （緑背景: 輝度88  → |88-88|=0    → 背景、無視）
4. そのコンテンツピクセルが新しいフレームでも同じ場所にあるか確認
5. 90% 以上残っていれば「追記されただけ」と判定
```

## 開発中の試行錯誤

### スーパーセット判定 ver.1（変化方向チェック）

最初の実装は「変化したピクセルがどちらの方向に変化したか」を見ていた。

```
暗くなった変化が多い → テキストが追加された（白背景スライド）
明るくなった変化が多い → チョークが追加された（黒板）
85% 以上が同方向 → 追記と判定
```

**問題点**: 変化量に上限（30%）を設けており、黒板の下半分まるごと追記されると
変化率が 30% を超えて「別スライド」と誤判定してしまった。

```
失敗例:
┌──────────┐       ┌──────────┐
│ 数式①    │       │ 数式①    │
│          │  →    │ 数式②    │  ← 変化率35% → 別スライド扱い（誤）
│          │       │ 数式③    │
└──────────┘       └──────────┘
```

### スーパーセット判定 ver.2（コンテンツ保持率 + 固定閾値）

「変化方向」ではなく「古いコンテンツが残っているか」に切り替えた。
変化量の上限も撤廃。

しかし「コンテンツ」の判定を**固定の輝度値（60以上）**で行っていたため問題が発生。

```
緑背景の輝度: 約88
コンテンツ閾値: 60
→ 88 > 60 → 緑の背景もコンテンツとして扱われてしまう

別スライドに変わっても緑背景は同じ → 保持率が高くなる → 同じスライドと誤判定
→ 全スライドが1クラスタになり、最後の1枚しかPDFに残らなかった
```

### スーパーセット判定 ver.3（コンテンツ保持率 + 中央値ベース）← 現在

背景色を「全ピクセル輝度の中央値」で動的に推定するように変更。

```
固定値（60以上 = コンテンツ）
    ↓
中央値との差（40以上外れた = コンテンツ）

緑背景(88)を中央値として推定 → 差=0 → 背景として除外 ✓
白チョーク(200) → 差=112 → コンテンツとして検出 ✓
```

これにより緑黒板・白板・スライドツールなど背景色を問わず正しく動作するようになった。
